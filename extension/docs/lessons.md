# Lessons — 반복 실패에서 얻은 원칙

> 코드 세부가 아니라 **다음에도 통하는 원칙**만 적는다. 지엽적 수치·클래스명은 금지 — 그건 일기지 교훈이 아니다.

---

## 1. 테스트 환경은 실제 환경을 그대로 닮아야 한다

**무슨 일**: 변경을 실제 앱이 아니라 손으로 만든 "축소판 테스트 환경"에서 검증했다. 시간이 지나며 그 축소판이 실제 앱과 **조금씩 어긋났다**. 결과적으로 테스트는 통과하는데 실제 앱에선 깨졌고, 사용자가 실제 화면을 직접 보고서야 발견했다. 같은 유형의 버그가 여러 번 반복됐다.

**왜 문제인가**: 실제와 다른 테스트는 "통과"가 거짓 안심을 준다. 없느니만 못하다 — 테스트가 없으면 의심이라도 하지만, 틀린 테스트는 믿어버리기 때문이다.

**원칙**:
- 테스트용 대역(sandbox·mock·stub)은 **실제 실행 환경을 최대한 그대로 재사용**한다. 설정·자원을 손으로 복제하면 반드시 drift한다.
- 부득이 근사(近似)한다면, **가장 먼저 "이 대역이 실제의 알려진 동작을 재현하는가"를 확인**한다. 현실을 재현 못 하는 대역은 변경을 검증할 자격이 없다.

**적용법**: 수정이 테스트에선 "통과"인데 사용자가 "여전히 깨진다"고 하면, 수정 자체를 다시 파기 전에 **테스트 환경이 실제와 다른 것부터 의심**하라. 대역을 실제와 최소 1회 대조해 두면 이 함정을 피한다.

---

## 2. "된다"와 "잘 보인다"는 다르다 — 기능 검증과 디자인 검수는 별개

**무슨 일**: 동작(기능)이 되는 것만 확인하고 넘겼더니, 시각적으로는 요소들이 크기·정렬·간격이 어긋나 있었다. 사용자가 눈으로 잡아냈다.

**원칙**: 상태가 바뀌는 UI는 **논리적 정상(무엇이 보이나)** 과 **시각적 정합(같은 크기·정렬·간격으로 보이나)** 을 따로 검수한다. 특히 두 표현(예: 편집 전/후)이 오갈 때는 **바뀌어도 튀지 않는지**를 픽셀 단위로 확인한다. 스크린샷 한 장 + 요소 실측이 "동작 로그"보다 많은 걸 말해준다.

**적용법**: 기능 통과 후 반드시 "한 세트로 읽히나? 전환 시 흔들리나?"를 별도로 본다.
